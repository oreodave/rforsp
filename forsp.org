** NOTE Observation on heap allocated objects
Pairs and Closures are the other two things we allocate, and they
happen to be identical in size (two pointers).

So if we create our own allocator, they're the only two objects we'd
need to be concerned about.  Naturally, as they're equivalently sized,
we don't need to care about padding our allocations out at all, or
having to tag them somehow in the allocator.

A standard old dynamic bump allocator would do the trick here, if we
didn't need to maintain pointers.  So arenas would be the next natural
fit.  However, even with an arena we're still leaving garbage around.
We need a way of collecting it, so we don't run out of memory for
really large programs.
** WIP Cheney's algorithm
Cheney's algorithm describes a method of allocation and collecting.
We maintain the heap as is, and when it overfills we enter
~collection~.  To collect, we consider the heap as two portions:
~from~ and ~to~.  ~from~ is where our objects currently lie, and ~to~
will be where we stick em.

We consider ~to~ to be a form of bump allocator so we'll use
~bump_ptr~ to represent the bump index.

*** TODO Forwarding pointers
We need a way to say that a given allocation has actually been moved
over to ~to~ space; this will allow us to stop the later BFS in cases
where memory has already been copied to ~to~ space.

We do this by introducing a "FORWARD_POINTER" object type within the
tagged pointer model.  It essentially just masks a pointer into ~to~
space.

This forwarding pointer object is stored in the first object slot of
any given allocation (pair or closure).  So to check if an object has
already been moved we just need to check the tag on the first object
of an allocation.

We'll also want functions to return the pointer to the forwarded cell
(i.e. the copied object) given an object pointer.
*** TODO Evacuate (move from ~from~ to ~to~)
Evacuate is a function that drives this entire collection routine.
Given a pointer ~p~, it works roughly like this:
- copy contents of ~p~ over to ~to~ space, incrementing ~bump_PR~.
  Call the new pointer ~p'~
- replace first object slot at ~p~ with a forwarding pointer to ~p'~

So, when we look at ~p~ currently (i.e. when looking at stuff that
references ~p~), checking the first field for a forwarding pointer
should be enough to say that ~p~ has been migrated.

**** TODO The initial sweep
We start by evacuating all the roots.  In Forsp our main roots will be
the ~stack~ and any ~comp~/~env~ parameters we get when doing a
recursive ~compute~.

This should leave us with a few forwarding pointers in ~from~ space,
and a couple allocations to ~to~ space.
***** TODO Root maintenance
A few more words regarding the ~comp/env~ parameters we get in a
~compute~ call.

We enter a new ~compute~ call every time we call a closure, which uses
its own ~comp~ and ~env~.  If we collect during ~compute~,
~comp~/~env~ will most likely point to garbage.  This is very bad.

To make the GC aware of the fact that we want updates on these
pointers, we need to maintain a ~root~ stack that stores pointers to
the things we want updated.  Essentially a stack of ~obj_t **~.

Then, during the initial sweep when we're evacuating the roots, we can
both scan this root stack to do so /and/ update the roots with the new
pointers over in ~to~ space.
**** TODO The BFS
Then, we scan through all objects in ~to~ space (which should be our
roots) to move all the still accessible memory located in ~from~ space
into ~to~ space.  We do this by maintaining a ~scan_ptr~ which starts
at the beginning of ~to~ space.  Certainly ~scan_ptr < bump_ptr~ by
initial evacuation.

The routine goes as so:
1) Look at the members of the object at ~scan_ptr~ (there should be
   exactly 2)
   - If the cells are currently in ~from~ space, evacuate them,
     incrementing ~bump_ptr~ accordingly.
   - If they're already present in ~to~ space, or they're an
     immediate, we don't need to do anything.
2) Bump ~scan_ptr~ to the next object in ~to~ space (16 bytes).
3) Repeat (1) until ~scan_ptr >= bump_ptr~

Once we escape the loop, we must have scanned everything accessible
from ~to~ space, which by virtue must be everything accessible in the
program from the roots.
**** TODO The sweep
Once we've evacuated everything necessary in ~to~ space, we:
- Update pointers in the ~compute~ stack for ~comp/env~
- Cleanup all of ~from~ space
- Swap ~to~ and ~from~ such that any new allocations will occur in
  ~to~ instead of ~from~.
** TODO Generational GC
Most objects die young.  This is certainly the case for closures,
which drive most of the fun in Forsp.

A generational GC essentially means we have a further partitioning by
/generation/ of heap space.  We have a ~nursery~, which is a sequence
of fresh allocations which we may maintain via Cheney's algorithm, and
collect often.  Any objects that survive a collection in the ~nursery~
are considered old, and thus can be relegated to the ~old age home~,
which experiences less frequent collections and can use less expensive
algorithms to maintain.

Might be worth considering if analysis suggests it's worth it.

What we really need is Big Integers.  Allocated on the heap, they
allow us to store and compute with arbitrarily wide integers.
