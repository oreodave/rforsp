** PROJ [#B] Better slot pools :bibop:
The current GC manages a single pool of fixed 16-byte slots, used for
pairs, closures, and vec metadata (the ~vec_t~ header).  Vec ~items~
buffers are ~calloc~/~realloc~/~free~'d via libc, which fragments the
heap and requires the ~vec_bits~ special case in sweep/stop.

Proposal: BiBOP (Big Bag of Pages).  Multiple pools, each with a
different slot size corresponding to a power-of-two number of ~obj_t*~
pointers (4, 8, 16, 32, ... 512).  The ~vec_t~ header stays in the
existing 16-byte pool (tagged ~TAG_VEC~ as before), but ~vec_push~
allocates the ~items~ buffer from the pool whose slot size best fits
the requested capacity.  On resize (doubling), the old ~items~ buffer
is immediately returned to its pool's free list and the new one
allocated from the next size up.

Pools beyond 512 slots (or any missing size class) fall back to a
managed ~malloc~ with an inline header (~size~ + ~mark bit~) so the GC
can still trace and reclaim them uniformly.

Benefits:
- No ~calloc~/~realloc~/~free~ for common vec sizes, so fewer libc
  calls, better locality, lower fragmentation.
- Immediate reuse on resize: old buffer goes straight to free list
  instead of waiting for a GC cycle.
- Pool memory is contiguous and page-aligned, so we have more
  cache-friendly GC scans.

Questions:
- How to find the sized-pool slot from an ~items~ pointer during GC
  mark?  Copying ~mimalloc~, we could align the chunk such that the
  header is at the base of the region boundary - then, we just need to
  clear off the lower N bits of the specific pointer to get the top of
  the chunk.
- Chunk layout for sized pools: a 32-byte-slot chunk has half the
  slots per chunk vs a 16-byte one.  Do we keep ~GC_CHUNK_SLOTS~ fixed
  (variable chunk size) or fixed chunk size (variable slots per
  chunk)?  Fixed chunk size is simpler for address arithmetic.
- The existing ~gc_find_chunk~ already linear-searches for meta-pool
  chunks.  Adding N more pools to search multiplies the cost
  proportionally.  Worth bundling with the [[*Better ~gc_find_chunk~]]
  work.

** PROJ [#A] Native recursive closures :rec:
Instead of relying on the Y-combinator to construct recursive
functions (see [[file:examples/factorial.fp]]), introduce a ~rec~
keyword that marks a closure as self-referencing.

Syntax:
#+begin_src rforsp
[$self $n ...] rec $fn
#+end_src
(which is essentially the same syntax we have for Y-combinator based
recursive closures currently).

The ~rec~ primitive pops a closure from the stack and returns it
wrapped in a new ~TAG_REC~ object.  When a ~TAG_REC~ is called (via
~eval_call~), the closure itself is pushed onto the value stack as an
additional first argument, fulfilling the ~$self~ binding that the
body expects.  That's it!

Benefits:
- Eliminates the Y-combinator overhead:
  - extra closure constructions without any optimisation
  - ~force~ call (~env-find~, ~eval_call~)
  - deeper cframe stack per recursive invocation - particularly if not
    already tail call style
- Tiny syntactical change for all example programs.

** TODO [#B] Design bytecode for vec-based closures :bytecode:
Compile closure body vectors (~vec_t~ of ~obj_t*~) into a flat
instruction array (~bc_prog_t~) with explicit opcodes, replacing the
tag-dispatch loop in ~eval~.

Each instruction is a packed triplet: ~{u8 opcode, i8 stack_delta, u64
operand}~.  The operand indexes into a constant pool (interned atoms,
pre-compiled sub-closures, numeric literals).

[[file:src/compute.c::static inline void eval(cframe_t
*cframe)][Current dispatch location]]

Opcodes:
- ~BC_PUSH_NIL~, ~BC_PUSH_NUM~, ~BC_PUSH_ATOM~, ~BC_PUSH_CLOS~ —
  push immediates from the constant pool or inline operand
- ~BC_QUOTE~ — push constant without evaluation (replaces the quote
  special case)
- ~BC_RESOLVE~ — ~env_find~ the atom at the given constant index
- ~BC_CALL~ — pop value from stack → ~eval_call~
- ~BC_TAIL_CALL~ — same but unconditional frame reuse
- ~BC_RET~ — pop current cframe
- ~BC_DROP~, ~BC_DUP~, ~BC_SWAP~ — stack manipulation
- ~BC_PUSH_SELF~ — push current rec closure (~TAG_REC~ bodies only)
- ~BC_TAIL_CALL_SELF~ — tail-recurse into self (rec bodies only)

The compiler (~bc_compile~) walks a ~vec_t~ body and emits ~bc_inst_t~
entries.  An optimisation pass then runs:
- Pre-resolve non-shadowed atoms to ~BC_PUSH~ (closure
  pre-computation)
- Inline known call chains
- Replace ~BC_RESOLVE~ + ~BC_CALL~ at end-of-body with ~BC_TAIL_CALL~
- Replace ~self~ resolution inside a ~TAG_REC~ body with
  ~BC_PUSH_SELF~
- Basic stack effect analysis via ~stack_delta~

** COMPLETE [#A] Reader bugs :gc:reader:
The current interpreter fails to execute
[[file:examples/loads-of-cons.fp]].  If we lower
~GC_THRESHOLD_DEFAULT~ to a small number (i.e. 16) all examples fail
to execute as well.

Behaviour:
- The execution failure is always from ~env_find~ (i.e. unable to find
  a key).

RCA:
- Initial debugging suggests the object parsed from the file is
  incomplete, which implies a read phase issue.
- Limited stack scan algorithm is responsible for marking objects in
  the machine stack for primitives and the read phase.
- The recursive ping-pong between ~read~ and ~read_list~ can make old
  function stacks out of reach of the limited stack scan when a
  ~gc_collect~ triggers, which means they are left unmarked and thus
  free to reclaim.
- Behaviour was mistakenly faulted to "too high GC pressure"
  previously, which lead to an overly high GC collection threshold.
  This hid the issues with the read phase by making collections only
  trigger /after/ the read phase during ~compute/eval~, which is well
  tested with regards to GC.

Solutions to fix this follow.

Lowest effort - just needs a flag on the ~state_t~ or ~gc_t~.  Won't
introduce a branch in the ~gc_alloc~ since we can just ~&&~ the flag

See the ~reader!gc-suppress~ branch for the implementation.

Issues:
- Really large files will definitely force a collection the first time
  we allocate in ~compute~, which means a massive slow down on the
  first compute iteration we do.
*** DONE Complete stack march
Get the base pointer right from ~main~ and make every stack scan a
complete one.  Blows up performance - or at least, I think it would.
May not?

NOTE: having a [[*Better ~gc_find_chunk~][better ~gc_find_chunk~]]

obj_t *optimize_closure_body(obj_t *body, obj_t *env);
#+end_src

Takes a vector body and the closure's captured env.  Returns a new
vector with resolvable atoms replaced.

The core algorithm is essentially as follows in a waterfall
match-style:
- Any triplet of the form ~quote <x> pop~ makes <x> a shadowed symbol
  from then on.  Copy it.
- Any triplet of the form ~quote <x> push~ where <x> has not been
  shadowed may be inlined to ~quote env_find(x, env)~ then copied.
  Otherwise, just copy as is.
- Any other ~quote <x>~ should be copied as is.
- Any bare instance of an atom:
  - Shadowed?  Copy as is.
  - Otherwise, do ~y := env_find(x, env)~ then:
    - If y is a callable, set it up via a ~TAG_CALL~ instance then
      copy that.
    - Otherwise, copy Y as is.

NOTE: [[file:src/state.c::obj_t *env_find(obj_t *env, obj_t
*key)][env_find]] does cause an abort if the key isn't present in the
environment, but based on the algorithm above any failure would match
with failures at actual ~eval~ time.
*** BACKLOG In-place replacement after first optimisation
Once a body is /fully/ optimised (every resolvable atom resolved, no
shadowed left as-is), we could write the optimised items into the
original body's ~items~ array and set an ~optimized~ flag (one bit
stolen from ~capacity~ or ~length~).

On subsequent closure-construction from the same body, the optimiser
checks the flag and copies without re-resolving.

Caveats:
- Need a field on ~vec_t~ for the flag.
- Only safe after full resolution (no shadowed atoms remain).
- Multiple closure constructions from the same reader-level vec (e.g.
  in a loop) benefit from this.

Path to implement:
1) Add ~bool optimized : 1~ field to ~vec_t~ (fits in the existing 16

2) In ~optimize_closure_body~, after successful resolution, set

   ~src->optimized = true~.
3) On re-entry, if ~src->optimized~ is true, short-circuit: return

*** BACKLOG Nested closure (child) inlining
Currently ~TAG_VEC~ items inside a body are skipped (copied as-is).
A recursive pass could inline them: when encountering a ~TAG_VEC~
item, call ~optimize_closure_body~ on it immediately, then wrap in
~TAG_CALL~.

This eliminates one ~make_clos~ + one ~optimize_closure_body~ per
inner closure per parent call — the parent body becomes
~[CALL(opt_child), ...]~ instead of ~[child_vec, ...]~.

Cost: recursive optimisation at parent-construction time (one-time).
Benefit: amortises across all calls to the parent.

*** BACKLOG Multi-pass optimisation
If a nesting optimisation produces new items that are themselves
atoms, a second pass could resolve them.  In practice, ~TAG_CALL~
items hide their inner values from the optimiser, so this is only
needed if child-closure inlining introduces new atoms into the parent
body.  Deferred.
** BACKLOG Cheney's algorithm :gc:

Cheney's algorithm describes a method of allocation and collecting.
We maintain the heap as is, and when it overfills we enter
~collection~.  To collect, we consider the heap as two portions:
~from~ and ~to~.  ~from~ is where our objects currently lie, and ~to~
will be where we stick em.

We consider ~to~ to be a form of bump allocator so we'll use
~bump_ptr~ to represent the bump index.

*** BACKLOG Forwarding pointers
We need a way to say that a given allocation has actually been moved
over to ~to~ space; this will allow us to stop the later BFS in cases
where memory has already been copied to ~to~ space.

We do this by introducing a "FORWARD_POINTER" object type within the
tagged pointer model.  It essentially just masks a pointer into ~to~
space.

This forwarding pointer object is stored in the first object slot of
any given allocation (pair or closure).  So to check if an object has
already been moved we just need to check the tag on the first object
of an allocation.

We'll also want functions to return the pointer to the forwarded cell
(i.e. the copied object) given an object pointer.
*** BACKLOG Evacuate (move from ~from~ to ~to~)
Evacuate is a function that drives this entire collection routine.
Given a pointer ~p~, it works roughly like this:
- copy contents of ~p~ over to ~to~ space, incrementing ~bump_ptr~.
  Call the new pointer ~p'~
- replace first object slot at ~p~ with a forwarding pointer to ~p'~

So, when we look at ~p~ currently (i.e. when looking at stuff that
references ~p~), checking the first field for a forwarding pointer
should be enough to say that ~p~ has been migrated.
*** BACKLOG Core algorithm
**** BACKLOG The initial sweep
We start by evacuating all the roots:
- The stack scan will take care of any in-flight objects in C-stack
  variables.
- The frame stack scan will take care of our ~comp~/~env~ pairs
- State stack and env are obvious.

This should leave us with a few forwarding pointers in ~from~ space,
and a couple allocations to ~to~ space.
***** BACKLOG Root maintenance
Both the stack scan and the frame stack scan need to update, since
evacuation moves pointers into ~to~ space.
**** BACKLOG The BFS
Then, we scan through all objects in ~to~ space (which should be our
roots) to move all the still accessible memory located in ~from~ space
into ~to~ space.  We do this by maintaining a ~scan_ptr~ which starts
at the beginning of ~to~ space.  Certainly ~scan_ptr < bump_ptr~ by
initial evacuation.

The routine goes as so:
1) Look at the members of the object at ~scan_ptr~ (there should be
   exactly 2)
   - If the cells are currently in ~from~ space, evacuate them,
     incrementing ~bump_ptr~ accordingly.
   - If they're already present in ~to~ space, or they're an
     immediate, we don't need to do anything.
2) Bump ~scan_ptr~ to the next object in ~to~ space (16 bytes).
3) Repeat (1) until ~scan_ptr >= bump_ptr~

Once we escape the loop, we must have scanned everything accessible
from ~to~ space, which by virtue must be everything accessible in the
program from the roots.
**** BACKLOG The sweep
Once we've evacuated everything necessary in ~to~ space, we:
- Update pointers in the ~compute~ stack for ~comp/env~
- Cleanup all of ~from~ space
- Swap ~to~ and ~from~ such that any new allocations will occur in
  ~to~ instead of ~from~.
