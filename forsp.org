* TODO Rework Forsp
This subtree manages the refactor of the Forsp language where we
replace certain linked list oriented operations of the language to
favour dynamic arrays instead.
** TODO Design a Stack based data structure to replace Pairs
Pairs are a very natural concept within Forsp.  However, I think we
can get away with replacing them in favour of a slightly more natural
data structure: dynamically sized stacks, backed by slab allocated
arrays.

Why?  I think having to run all across the heap for linked lists is a
real pain.  As program sizes get larger, this only becomes more of an
issue.  In comparison, a dynamically sized stack at least allows us to
iterate a bit more seamlessly through a first depth level of
instructions.

Also consider the universal stack, which is used for function
parameters and returns.  A generalisable stack which user code can
generate and utilise would fit naturally here.
*** TODO Tagged pointer
We need two data types:
- ~stack~ which has ownership of the underlying data.
- ~stack_view~ which doesn't have ownership.

NIL should map to an empty stack.
*** TODO User API
Normal object based API, same as any other:
- ~stack-push~ takes an element from the universal stack, and a stack
  object, and pushes that element onto that stack object.  Stack
  object is preserved.
- ~stack-pop~ takes a stack object and pops an element off it, pushing
  that element onto the universal stack.  Stack object is preserved
  behind the freshly pushed item.
- ~stack-length~ takes a stack object and pushes the length of it onto
  the universal stack.
*** WAIT Stack frames
I'd really like to have some kind of notion of ~stack frames~ to
partition function calling.  Essentially, one can construct a stack
and swap it with the current calling stack.  The idea is that future
calls utilising the universal stack will instead push/pop on this
allocated one instead.

At the very least, it should make constructing toy stacks really easy.
Consider the task of creating a stack with 3 elements:
#+begin_src forsp
(
  nil 3 stack-push 2 stack-push 1 stack-push print
)
#+end_src

With the frame based API:
#+begin_src forsp
(
  nil push-frame
      3 2 1
  pop-frame
  print
)
#+end_src

... where essentially:
- ~push-frame~: given current running stack A, and stack object B on
  top of stack A: push A onto a /frame stack/, make B the new running
  stack
- ~pop-frame~: given current running stack B, and A as the top of the
  frame stack: pop A off the frame stack, make A the current running
  stack, push B onto the current running stack.

Environments are treated the same across frame borders:
#+begin_src forsp
(
  nil push-frame
      (2 *) $double
      3 2 1
  pop-frame
  print
  3 double print
)
#+end_src
** TODO Redesign reader
The reader should, instead of returning ~obj_t*~, push parsed ~obj_t*~
onto a stack it takes as parameter.

There are a few benefits to this:
- No need for a /read stack/ since we can just push the relevant
  pieces directly onto the parameterised stack.
- ~read~'s return value is free to be used as an error value of some
  kind, which we can propagate up the stack for better errors.

Furthermore, any instances of ~()~ should use stacks instead of linked
lists.  This will be crucial for later redesigns.
** WAIT Redesign environments
This is a hard one - environments as linked lists make perfect sense

shame, but this may have to wait till IR phase.
** TODO Redesign closures and compute
Instead of being linked lists of instructions, make closures
essentially stacks of instructions.  ~closures~ simply store a
sequence of instructions with their initial environment when
constructed, so a stack makes obvious sense here.

Since ~compute~ essentially linearly iterates through a linked list of
instructions, this is no different from running linearly through a
stack of instructions.  Furthermore, as ~read~ already very helpfully
produces a stack of instructions to interpret, this fits naturally.
* TODO Implement a IR phase
This subtree manages the construction and implementation of an IR
phase within the Forsp interpreter.

The big idea behind a IR phase is to make ~compute~ more performant.
We'll compile our super high level Forsp code into a sequence of
simpler instructions, which on their own should be super easy to run
through at mach speed.

The IR standard we define, along with the analogous ~compute~,
describe a VM.  A few instrinsics in this VM should map to the special
operators of the higher level Forsp language (quote ('), pop ($), push
(^)).  In a sense, this IR will represent the /kernel/ of the Forsp
language itself.

This is going to require some level of language redesign, and will
have multiple components.
** TODO Design IR
We'll first consider the ~compute~ function to motivate the
construction of certain instructions, then consider data types.
*** Data types
**** Closure rework
Instead of /linked lists/ being the method by which we create
closures, let's use dynamic arrays of IR instructions.  This does add
some overhead to small closures, but in trade we get much faster
compute times since we're iterating linearly through it.  Also the
Forth like components of Forsp (stack based computation) fall quite
naturally into array based compute.

This also unifies the notion of the runtime stack (which we can
implement trivially as a dynamic array) with the actual code we're
interpreting; if we introduce the right primitives, we could generate
code at runtime on the /compute stack/ that the interpreter will run
straight away.
**** Pair -> Stack rework
Since pairs aren't the fundamental /block making/ data type (instead
we're now using arrays), I want to extricate them as the main
container type as well.  Instead, let's use dynamic arrays as stacks!

Essentially we trade ~cons,car,cdr~ for the following:
- ~push(T, S)~: push the value T onto the stack S
- ~top(S)~: get the topmost value of stack S
- ~rest(S)~: get a view of the stack S excluding the topmost value

To get back some of the niceties of linked lists we'll use /stack
views/ along with /copy on write/ to make this work:
- ~rest(S)~ generates a /stack view/ which doesn't copy the underlying
  data, instead constructing a new array header with a pointer to the
  same data as S itself.  ~top~ works just fine with this.
- ~push(T, S)~ will copy iff S is a stack view, and mutate if S is a
  normal stack.

We type stacks as so:
- ~[]~ represents the empty stack
- ~[T]~ represents a stack with one item (type T)
- ~[...U T]~ represents a stack with the topmost item of type T, and
  the rest encoded with a container type U

So, for example, a stack with an integer topmost, then a symbol, then
an empty stack, would look like so ~[[] symbol integer]~.
*** Instructions
We want our IR to essentially make the ~compute~ function easier.
Let's break down the current implementation of the ~compute~ function:
- Given a block of code ~xs~ (currently implemented as a linked list),
  we iterate through every element ~x_n~
- Each element ~x_n~ is distinguished by tag:
  1) If ~x_n~ is an atom then:
     1) If it's a quote, push ~x_{n+1}~ literally, and move to ~x_{n+2}~
     2) Otherwise, lookup that atom in the current environment to get
        ~y~, then:
        1) If ~y~ is a closure or primitive, call it
        2) Otherwise, push whatever ~y~ is on the stack
  2) If ~x_n~ is a linked list (NIL or otherwise), make a closure out
     of it, binding the current environment
  3) If ~x_n~ is any other type (number, closure, primitive), push it
     onto the stack

We'll break down these paths and consider relevant instructions we'd
need in our IR to represent them.
**** 1 - Atoms
***** 1.1: Quoted atoms
This only occurs when one of the special operators is used, as the
reader compiles them into multiple operations:
- ~'x~ where ~x~ is an atom compiles to ~quote x~
- ~$x~ where ~x~ is an atom compiles to ~quote x pop~
  - This takes a value off the stack and binds it to the symbol ~x~ in
    the current environment.  ~pop~ is implemented as a primitive.
- ~^x~ where ~x~ is an atom compiles to ~quote x push~
  - This looks up ~x~ in the current environment and pushes the
    relevant value onto the stack.  ~push~ is implemented as a
    primitive.

See how they all manipulate an atom?  We can make this specific by
making them into exact instructions:
- ~'x~ maps to ~PUSH_SYMBOL(x)~.  This symbol is pushed onto the stack
  as is.
- ~$x~ maps to ~BIND(x)~ where ~BIND~ pops a value already on the
  stack
- ~^x~ maps to ~LOOKUP(x)~ where ~x~ should already have an associated
  value in the environment.  The associated value is pushed onto the
  stack.
***** 1.2 Atom calling
This is a core part of the Forsp language: raw unquoted symbols
compile to function calls.  Given [[*1.1: Quoted atoms][1.1]] has the
idea of ~LOOKUP(x)~, we can either go for:
- ~LOOKUP(x)~ then ~CALL~ which:
  - Pushes the associated value of X onto the stack
  - ~CALL~ then pops that value and performs a "call" depending on the
    type of the value (closures, primitives -> compute, otherwise ->
    push value)
- ~CALL(X)~ which embeds the symbol into the call directly.

Given ~BIND(x)~ expects values on the stack, it might be better to
have a 0-arity ~CALL~ (i.e. the former approach).  Furthermore, this
allows us to extract away the retrieval mechanisms for every type of
callable (see [[*3 - Normal pushes][3]]) so ~CALL~ is purely about
actually doing a call.

**** 2 - Blocks of Code
When a linked list (which is the current notion for "block of code")
is encountered, we generate a closure and push the associated value
onto the stack.  Here's where the [[*Redesign reader][reader
redesign]] (which see) will be really important - instead of leaving
it to ~compute~ time to generate the closure, we can do it ahead of
time at the ~read~.

The main things to consider for IR design is:
- ~MAKE_CLOSURE(idx)~ where ~idx~ is some way of pointing to the raw
  code pertaining to the closure within the IR, pushing a closure
  object onto the stack.
  - ~MAKE_CLOSURE~ also needs to bind the current environment and
    store it into the pushed closure object.
**** 3 - Normal pushes
When a general object that isn't already caught by the previous
conditions is encountered, push it onto the stack.  For the sake of
analysis I think it's better if we have unique PUSH constructors for
each type:
- ~PUSH_NUM(X)~ where X is a number
- ~PUSH_CLOSURE(X)~ where X is a closure object (unlikely to be used
  in first pass stuff)
- ~PUSH_PRIMITIVE(X)~ where X is a primitive
**** Primitives
Given how small the number of primitives required for self
implementation actually are, we may as well embed them into IR itself
directly, which should allow the reader to elide a bunch of
~PUSH_PRIMITIVE(x) CALL~ instances.
**** Summary
So in total we have the following extra instructions:

** TODO Stack effect/Type checking
Forsp currently lacks stack effect analysis and type checking.  It's
not needed for a minimalist implementation (hence why it's not present
in the original), but these are several reasons this would be a good
idea:
- having a formal understanding of the exact effects (stack effects,
  along with type pre-requisites) should help with catching a class of
  reasoning/soundness errors
- will help with optimisation passes
- good exercise in verifying the core kernel of the language itself
